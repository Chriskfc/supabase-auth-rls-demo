The wire what the database sent

The query below is identical for both accounts. It is the only line of data-fetching code in this file.

await supabase
  .from('quotes')
  .select('*, customers(name), quote_items(*)')
  .order('created_at')

No filter. No where owner_id = …. Nothing in the front end decides who sees what.

The policy doing it 
-- runs inside Postgres, on every single query
create policy "quotes: rep reads own, admin reads all"
  on public.quotes for select
  to authenticated
  using (owner_id = auth.uid() or public.is_admin());

Withheld rows are not hidden, greyed out, or filtered in JavaScript. They were never put on the wire. Open the network tab and look — they are not in the response.

Writes are checked too using vs with check

Reading is only half of it. The activity log accepts an entry from any staff member — but only in their own name. The second button tries to write an entry as Dana. The database refuses it, and the raw error is printed exactly as it comes back.

Table with row level security switched off

You are not signed in to this part. You are reading it anyway.

The key this page uses is public — it ships inside the HTML, and it is meant to. Row level security is what makes that safe. This one table was left without it, and so anybody who opens the page, signed in or not, gets every row. This is the accident the policies above prevent.
